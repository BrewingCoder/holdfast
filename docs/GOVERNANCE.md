# HoldFast Governance

## Project Structure

HoldFast is a community-driven, open-source project. There is no company behind it, no paid tier, and no commercial support. It exists because the upstream project (Highlight.io) was shut down and the community wanted to keep it alive.

## Maintainership

This project is maintained on a **best-effort basis**. The maintainer reviews PRs, triages issues, and merges contributions when time permits — but makes no commitment to response times or guaranteed availability.

If a PR sits for a while, it's not a statement about its quality. A polite ping is always welcome.

## Community Support

The community is the primary support channel. If you have a question or run into a problem:

1. Search existing issues first
2. Open a new issue with clear reproduction steps
3. Help others when you can — this is how the project stays healthy

There is no dedicated support team. Users helping users is the model.

## Contributions

All contributions are welcome — bug fixes, features, documentation, testing. To contribute:

1. Fork the repo and create a feature branch
2. Keep changes focused — one concern per PR
3. Include context in your PR description (what and why)
4. Be patient with review — it happens on a best-effort basis

By submitting a PR, you agree that your contribution is licensed under AGPL-3.0, consistent with the project license.

### Agentic Contributions

Contributions generated by AI coding agents (Claude Code, Copilot, Cursor, Aider, etc.) are fully supported and welcome. The repository is seeded with documentation specifically designed to help both human and agentic contributors work effectively:

- **`CLAUDE.md`** — Top-level project context: repository structure, architecture overview, build commands, development workflow, testing strategy, and key conventions
- **`HOLDFAST-NOTES.md`** — Fork-specific context: what was removed from upstream, what was changed, current state of the codebase, and roadmap
- **`CHANGELOG-FORK.md`** — Detailed record of every modification from the upstream Highlight.io codebase, including file-level change descriptions, stubbed functions, and removed environment variables
- **`GOVERNANCE.md`** — This file: project rules, contribution expectations, and license requirements
- **In-code documentation** — GraphQL schemas, Go struct tags, TypeScript types, and test files serve as living documentation of interfaces and expected behavior

These files are maintained as first-class project artifacts. If you are building tooling or agents that interact with HoldFast, these are your entry points.

**Agentic PRs are held to the same standard as human PRs.** All contributions — regardless of origin — are reviewed before acceptance. Specifically:

- Code must compile and pass existing tests
- Changes must be focused and well-scoped (no drive-by refactors)
- PR descriptions must explain what changed and why
- Generated code must not introduce security vulnerabilities, hardcoded secrets, or license-incompatible dependencies
- Bulk reformatting, unnecessary abstraction, or speculative "improvements" will be rejected
- Agent-generated PRs should disclose that they are agent-generated (a one-liner in the PR description is sufficient)

## Moderation

Issues and PRs that are disruptive, abusive, or off-topic may be closed without discussion. This is a small project with limited maintainer bandwidth — keeping the signal-to-noise ratio high is essential.

## Decision Making

For day-to-day changes, the maintainer has final say. For larger architectural decisions, discussion in issues is encouraged so the community can weigh in before work begins.

## Future Governance: HoldFast Contributor Board

If the project reaches sufficient community size and sustained contribution activity, a **HoldFast Contributor Board** will be formed to share governance responsibilities. The board would:

- Set roadmap priorities and release milestones
- Review and approve significant architectural changes
- Resolve disputes on contested PRs or design decisions
- Manage community health and moderation policy
- Represent diverse perspectives from the contributor and user base

Board members would be appointed from active contributors based on sustained, quality participation — not volume. The criteria and process for board formation will be defined transparently when the time comes.

### Non-Negotiable Constraints

Regardless of governance structure — now or in the future — certain things are not up for debate:

- **License remains AGPL-3.0.** The project will not be relicensed. Period. No CLAs that grant relicensing rights. No "dual license" carve-outs for commercial use. AGPL-3.0 is the deal.
- **No reintroduction of SaaS billing, marketing, or telemetry-home code.** HoldFast is a clean OSS project. Features that exist to monetize users or report usage to third parties will not be accepted.
- **No proprietary dependencies.** The project must remain fully self-hostable using only open-source infrastructure. Integrations with third-party services (Slack, Jira, etc.) are fine — hard dependencies on proprietary services for core functionality are not.

This governance model is intentionally simple. It will evolve if and when the project needs it — but the constraints above are permanent.
